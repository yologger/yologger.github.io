들어가며

이전 포스트에서는 중위 함수(Infix Function)에 대해 알아봤습니다. 이번 포스트에서는 인라인 함수(Inline Function)와 noinline, crossline에 대해 알아보겠습니다.

인라인 함수

함수 앞에 키워드 inline을 붙이면 함수를 호출하는 곳에 함수의 몸체를 그대로 복사합니다. 우선 inline을 사용하지 않은 일반 함수 예문을 보겠습니다.

그럼 키워드 inline를 붙이면 어떤 일이 발생할까요?

아래 코드처럼 doSomething()메소드의 몸체를 그대로 복사하고 있습니다.

그렇다면 inline 함수는 왜 쓰는걸까요?

Runtime Penalties

코틀린에서 함수를 함수의 인자로 전달할 때 문제가 발생합니다. 입니다. 함수를 함수의 인자로 전달하면 추가적인 메모리 할당이 발생하기 때문입니다.

위 코드를 아래와 같이 자바로 변환하면 someMethod()메소드를 호출하기 위해 객체를 생성합니다.

객체를 생성한다는 것은 메모리를 그만큼 사용한다는 것입니다. 만약 이 함수를 열 번 호출하면 열 개의 객체가 생성되며, 이러한 부분이 성능을 떨어뜨릴 수 있습니다.

이제 함수 앞에 키워드inline을 붙여보겠습니다.

위 코드를 컴파일하면 객체를 생성하지 않고 코드 자체를 호출하는 부분으로 복사합니다.

이처럼 inline 함수를 사용하면 메모리 성능을 향상시킬 수 있습니다. 하지만 코드 양이 많은 함수를 inline 함수로 사용하면, 컴파일된 코드의 양도 많아질 수 있습니다. 따라서 inline 함수는 1~3줄 정도의 함수에 사용하는 것을 권장합니다.

주의할 점

inline 함수는 내부적으로 코드를 복사하기 때문에, 인자로 전달받은 함수를 다른 함수의 인자로 전달하거나 다시 참조할 수 없습니다.

위 코드는 다음과 같은 컴파일 에러를 발생시킵니다.

noinline

그럼 위 예제에서 func2()만 접근이 가능하고 func2()함수만 다른 함수의 인자로 전달되도록 할 수는 없을까요? 바로 이러한 경우 키워드 noinline를 사용합니다. 키워드 noinline붙은 키워드는 컴파일 시 일반 함수와 동일하게 동작합니다. 따라서 다시 함수에 접근하거나 다른 함수의 인자로 전달할 수 있습니다.

crossline

키워드 crossline을 이해하려면 우선 non-local returns에 대해 알아야 합니다. 아래 코틀린 코드를 살펴봅시다.

이제 위 코드를 디컴파일하면 다음과 같은 자바 코드를 확인할 수 있습니다.

디컴파일된 코드에 System.out.print("doSomething end")가 포함되지 않았다는 것을 확인할 수 있습니다. 함수의 인자로 전달된 함수 내부에서 return을 호출했으므로 doSomething()함수 자체가 종료된 것입니다. 이러한 현상을 non-local returns라고 합니다.

crossline은 이러한 문제를 해결할 때 사용합니다.

앱 게시 취소

앱 게시가 취소되면 새로운 사용자는 플레이스토어에서 앱을 검색하거나 다운로드할 수 없습니다. 그러나 기존 사용자는 앱을 계속 사용할 수 있습니다.

우선 구글 플레이 콘솔(Google Play Console)에 접속합니다.

왼쪽 사이드바 메뉴 > All apps로 이동합니다. 이 곳에서 출시한 앱 목록을 확인할 수 있습니다. 삭제할 앱을 선택합니다.

Setup > Advanced settings로 이동합니다.

App availability를 Unpublished로 변경하고 Save changes버튼을 누르면 앱 게시가 취소됩니다.

변경 사항이 반영되는데 몇 시간이 걸릴 수 있습니다.

게시가 취소된 앱은 삭제되기 전까지 다시 출시할 수 있습니다.

앱 삭제

구글 플레이 콘솔 고객센터(Google Play Console Help)에 접속합니다.

지원이 필요한 사항을 입력합니다.

앱 삭제를 선택합니다.

이메일을 선택합니다.

충족 사항을 확인합니다.

이름, 성, 이메일 주소, 내 위치, 개발자 이름을 입력합니다.

개발자 계정 ID를 입력합니다. 계정 세부정보 페이지를 선택하면 개발자 계정 ID를 확인할 수 있습니다.


패키지 이름을 입력합니다.

패키지 이름은 구글 플레이 콘솔 > All apps에서 확인할 수 있습니다.

문제 해결에 도움이 되는 스크린샷을 추가합니다. 삭제할 앱의 스크린샷을 첨부합니다.

제출하면 처리되는데 1~2일 정도가 소요됩니다.

[코틀린] 리플렉션(Reflection)

2019/02/07 카테고리 Kotlin

코틀린(Kotlin)에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

리플렉션(Reflection)은 런타임에서 구체적인 클래스 타입을 몰라도 그 클래스의 멤버변수, 메소드 등에 접근할 수 있도록 하는 API입니다.

Class와 KClass

우선 코틀린의 리플렉션을 이해하려면 자바의 리플랙션에 대해 알아볼 필요가 있습니다. (자바의 리플랙션은 이 포스트를 참고하시기 바랍니다.)

자바에서는 클래스에 대한 정보를 Class클래스에 저장할 수 있습니다. 이를 클래스 참조(Class Reference)라고 하며, 클래스를 참조하는 방법에는 여러 방법이 있습니다.

반면 코틀린에서는 더블콜론(::)을 사용하여 클래스에 대한 정보를 가져올 수 있습니다.

주의할 점은 Class클래스가 아닌 KClass라는 클래스에 정보가 저장된다는 점입니다. 이는 자바에서 사용하는 클래스와 코틀린에서 사용하는 클래스가 다르기 때문입니다.

코틀린의 Kclass를 자바의 Class로 변환하려면 .java를 뒤에 붙여주면 됩니다.

(3) 출시

[코틀린] 접근 제한자(Access Modifiers)

2019/02/06 카테고리 Kotlin

코틀린(Kotlin)에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

접근 제한자

접근 제한자(Access Modifiers)는 외부에서 클래스 내부의 메소드나 멤버변수에 접근하는 것을 제한하는 것입니다. 코틀린은 네 개의 접근 제한자를 지원합니다.

(1) private

private이 붙은 멤버변수, 메소드, 생성자는 클래스 내부에서만 접근할 수 있습니다. 클래스에는 붙일 수 없습니다.

private으로 선언된 변수는 상속받는 자식 클래스에서도 접근이 불가능합니다.

(2) protected

protected가 붙은 멤버변수, 메소드, 생성자는 해당 클래스와 자식 클래스에서만 접근할 수 있습니다. 클래스에는 붙일 수 없습니다.

(3) internal

internal이 붙은 멤버변수, 메소드, 생성자는 같은 모듈 안의 어디에서든 접근할 수 있습니다. 이 접근 제한자는 클래스 앞에도 붙일 수 있습니다. 코틀린 공식 문서에서 말하는 같은 모듈은 아래 상황을 뜻합니다.

  1. Android Studio Module
  2. IntelliJ IDEA Module
  3. Maven Project

쉽게 말하면 안드로이드 스튜디오 프로젝트의 같은 모듈에서는 접근이 가능하다고 보면 되겠습니다.

(4) public (default)

접근 제한자를 따로 붙이지 않으면 이 접근 제한자가 적용됩니다. public이 붙은 멤버변수, 메소드, 생성자는 다른 모듈에서도 접근할 수 있으며, 클래스에도 붙일 수 있습니다.

마치며
