[안드로이드] compileSdkVersion, minSdkVersion, targetSdkVersion 차이점

2019/03/06 카테고리 Android, Android Framework

들어가며

이전 포스트에서는 런타임에서 SDK 버전을 확인하는 방법에 대해 알아봤습니다. 이번 포스트에서는 compileSdkVersion, minSdkVersion, targetSdkVersion의 차이점에 대해 알아보겠습니다.

(1) compileSdkVersion

안드로이드 스튜디오에서 앱을 빌드할 때 어떤 SDK 버전을 사용할 것인지 지정합니다. 즉 comileSdkVersion은 지원할 수 있는 가장 높은 버전을 의미합니다.


안드로이드 스튜디오의 SDK Manager에 해당 SDK가 설치되어있어야 합니다.

(2) minSdkVersion

최소 지원 SDK 버전을 의미합니다. 즉 해당 버전 이하의 안드로이드 운영체제가 설치된 기기에서는 앱을 설치할 수 없습니다.

(3) targetSdkVersion

앱을 실제 기기에서 실행할 때 어떤 SDK 버전을 사용할지 지정합니다. 기능은 똑같아도 SDK 버전에 따라 다르게 동작하는 API도 있습니다. 이때는 targetSdkVersion에 맞는 API를 사용하게 됩니다.

targetSdkVersion을 지정하지 않으면 기본 값은 minSdkVersion과 동일하게 적용됩니다.

예제

모듈 수준의 build.gradle파일에서 다음과 같이 SDK 버전 설정을 설정합니다.

이는 SDK 버전이 25 이상 30 이하에서만 이 앱이 실행될 수 있음을 의미합니다. 또한 실제 동작은 27 버전으로 동작하게 됩니다. 보통 실무에서는

minSdkVersion <= targetSdkVersion == compileSdkVersion

로 SDK 버전을 설정합니다.

[안드로이드] 안드로이드 SDK란?

2019/03/04 카테고리 Android, Android Framework

안드로이드(Android)에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

이전 포스트에서는 디컴파일러(Decompiler)에 대해 알아봤습니다. 이번 포스트에서는 안드로이드 SDK에 대해 알아보겠습니다.

안드로이드 SDK

안드로이드 SDK(Software Development Kit)는 안드로이드 앱을 개발하기 위한 툴 킷입니다.

안드로이드 스튜디오는 SDK Manager를 제공하는데 이 곳에서 안드로이드 SDK를 설치, 삭제, 설정 할 수 있습니다. 또한 SDK Manager에서 SDK의 설치 경로를 확인할 수 있습니다.

그럼 터미널에서 SDK의 설치 경로로 이동해보겠습니다.

SDK에 무엇이 설치되어있는지 확인해봅시다.

(1) system-images

system-images폴더에는 에뮬레이터가 사용하는 안드로이드 운영체제 이미지가 있습니다.

(2) platforms

platforms폴더에는 안드로이드 플랫폼이 설치되어 있습니다.

(3) tools

tools폴더와 tools/bin폴더에는 앱을 개발하고 디버깅하기 위한 여러가지 툴이 있습니다. 에뮬레이터, SDK 매니저, AVD 매니저, UI 뷰어 등이 이 폴더에 설치되어있습니다.

안드로이드 스튜디오에서도 tools폴더에 설치된 툴을 확인할 수 있습니다.

(4) build-tools

build-tools폴더에는 앱을 빌드하기 위한 도구들이 설치되어있습니다.

(5) platform-tools

platform-tools폴더에는 안드로이드 기기와 연결했을 때 사용하는 도구들이 설치되어있습니다. adb, fastboot, systrace, sqlite3 등이 이 폴더에 설치되어있습니다.

마치며

이번 포스트에서는 안드로이드 SDK에 대해 알아봤습니다. 다음 포스트에서는 런타임에서 SDK 버전을 확인하는 방법에 대해 알아보겠습니다.

[안드로이드] JVM, 달빅 가상머신, ART의 차이점

2019/03/02 카테고리 Android, Android Framework

안드로이드(Android)에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

이전 포스트에서는 DEX 파일과 APK 파일에 대해 알아봤습니다. 이번 포스트에서는 JVM, 달빅 가상머신, ART에 대해 알아보겠습니다.

JVM

JVM(Java Virtual Machine)은 자바 바이트코드를 실행하는 가상머신입니다.

자바 컴파일러는 자바 소스코드를 컴파일하여 자바 바이트코드를 생성합니다. JVM은 이 자바 바이트코드를 실행하는 주체입니다. 플랫폼에 적절한 JVM만 설치되어있으면 어떤 플랫폼에서든 자바 바이트코드를 실행할 수 있습니다.

DVM

DEX 컴파일러는 자바 바이트코드를 DEX(Dalvik Executable) 파일로 통합합니다.

DVM(Dalvik Virtual Machine)은 안드로이드 애플리캐이션을 실행하는 주체입니다. 정확하게 말하자면 DVM는 이 DEX 파일을 실행합니다.

구글은 자바 가상머신인 JVM을 사용할 수도 있었지만 오라클과의 라이센스 문제 때문에 DVM을 개발하여 안드로이드에 넣게 됩니다.

뿐만 아니라 모바일 기기는 베터리 수명, 컴퓨팅 능력, 메모리 등의 자원이 데스크탑 환경에 비해 열악합니다. 이 때문에 모바일 기기에 맞게 나온 가상머신이 바로 DVM 입니다.

DVM은 내부적으로 JIT(Just In Time) 컴파일러를 사용합니다. 이 컴파일러는 애플리캐이션을 실행할 때마다 자바 바이트코드를 기계어로 컴파일합니다. 이 덕분에 설치 파일은 작아졌으나 CPU, 메모리, 베터리 사용이 높아지게 되었습니다.

ART

ART(Android Runtime)은 안드로이드에서 사용되던 기존 DVM의 한계점을 해결하기 위해서 구글에서 새로 개발한 런타임입니다.

ART는 내부적으로 AOT(Ahead On Time) 컴파일러를 사용합니다. 이 컴파일러는 애플리캐이션을 설치할 때 자바 바이트코드를 기계어로 컴파일합니다. 따라서 설치 파일이 커지고 설치 시간이 오래 걸린다는 단점이 있으나 런타임에서 CPU, 메모리, 베터리 사용량이 낮아지고 퍼포먼스가 좋아지게 되었습니다.

이전 포스트에서는 앱 번들(App Bundle)에 대해 알아봤습니다. 이번 포스트에서는 DEX 파일과 APK 파일에 대해 알아보겠습니다.

APK

APK(Android Application Package)는 안드로이드 어플리캐이션 설치 파일입니다. 안드로이드 시스템에서 APK 파일을 실행하여 어플리케이션을 설치할 수 있습니다. Windows 계열의 운영체제에서 인스톨러 역할을 한다고 보시면 됩니다.

빌드 프로세스

APK 파일은 다음과 같은 과정을 거쳐 생성됩니다.

  1. 안드로이드 프로젝트는 소스코드 및 리소스로 구성된 모듈과 의존성으로 구성됩니다.
  2. 그래들는 이 모든 것을 빌드하여 DEX 파일과 컴파일된 리소스를 생성합니다.
  3. APK Manager는 개인키로 서명하여 APK 파일를 생성합니다.
  4. APK 파일은 구글 플레이스토어에 업로드하거나 안드로이드 시스템에 직접 설치할 수 있습니다.
