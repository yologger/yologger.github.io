[자바] 람다식(익명함수)와 함수형 인터페이스(Functional Interface)

2018/03/16 카테고리 Java

자바(Java)에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

이전 포스트에서는 일급 객체, 함수형 프로그래밍에 대해 알아봤습니다. 이번 포스트에서는 람다식(익명함수), 함수형 인터페이스에 대해 알아보겠습니다.

람다식

람다식(Lambda Expression)은 함수를 하나의 식(expression)으로 표현한 것입니다. 함수를 람다식으로 표현하면 메소드의 이름이 필요 없기 때문에 익명 함수(Anonymous Function)라고도 불립니다.

코틀린에서의 람다식

코틀린(Kotlin), 스위프트(Swift)처럼 비교적 최근에 개발된 프로그래밍 언어는 자체적으로 람다식을 지원합니다. 그럼 코틀린 먼저 예제를 살펴보겠습니다. 보통 코틀린에서 함수는 다음과 같이 선언합니다.

자바에서의 람다식

자바는 코틀린, 스위프트 같은 언어에 비해 비교적 오래되었습니다. 또한 객체지향 프로그래밍을 패러다임으로 만들어졌기 때문에 람다식과 함수형 프로그래밍을 지원하지 않았습니다. 그러나 2014년 발표된 Java 8부터 함수형 프로그래밍을 지원하기 위한 자바 API가 추가되었습니다.

아쉽게도 자바는 함수 타입이 없으므로 람다식을 변수에 저장하거나 매개변수로 전달할 수 없습니다. 이를 해결하기 위해 도입된 개념이 @FunctionalInterface어노테이션 입니다.

@FunctionInterface

함수형 인터페이스를 구현하기 위해서는 인터페이스를 정의하고 그 내부에는 한 개 뿐인 추상 메소드를 선언하고, @FunctionalInterface 어노테이션을 붙여주면 됩니다.

이제 마치 변수에 함수를 할당하는 것처럼 사용할 수 있습니다.

자바 API에서는 자주 사용될 것 같은 함수형 인터페이스가 이미 정의되어있습니다.

(1) Supplier

매개변수 없이 반환값만 갖는 함수형 인터페이스입니다. Supplier는 T get()을 추상 메소드로 갖고 있습니다.

(2) Consumer

매개변수가 있고 반환값은 없는 함수형 인터페이스입니다. Consumer는 void accept(T t)를 추상메소드로 갖고 있습니다.

(3) Function <T, R>

매개변수도 있고 반환값도 있는 함수형 인터페이스입니다. Function은 R apply(T t)를 추상 메소드로 가지고 있습니다.

(4) Predicate

매개변수가 있고 boolean타입의 값을 반환합니다. Supplier는 boolean test(T t)을 추상 메소드로 갖고 있습니다.

안드로이드에서의 람다식

람다식은 안드로이드 어플리케이션 개발에서도 자주 사용합니다. 보통 버튼같은 위젯을 클릭했을 때 실행할 구문을 람다식 안에 구현합니다.

우선 람다식을 사용하지 않고 익명 클래스를 사용하여 특정 버튼을 클릭했을 때 실행할 구문을 등록해보겠습니다.

데이터 타입 생략도 가능합니다.

함수의 몸체가 한 줄일 때는 중괄호{}도 생략하며, 한 줄로 표현할 수 있습니다.

매개변수가 한 개일 때는 소괄호()도 생략할 수 있습니다.

[자바] 일급 객체, 함수형 프로그래밍

2018/03/15 카테고리 Java

자바(Java)에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

이전 포스트에서는 어노테이션(Annotation)에 대해 알아봤습니다. 이번 포스트에서는 일급 객체, 함수형 프로그래밍에 대해 알아보겠습니다.

일급 객체 (First-class citizen)

함수형 프로그래밍을 공부하다보면 일급 객체란 개념이 많이 언급됩니다. 다음 세 가지 조건을 모두 충족하면 일급 객체라고 할 수 있습니다.

  1. 변수나 상수에 할당할 수 있어야 한다.
  2. 함수의 인자로 전달할 수 있어야 한다.
  3. 함수의 반환값으로 반환할 수 있어야 한다.

코틀린에서 함수는 일급 객체이지만 자바에서 함수는 일급 객체가 아닙니다. 우선 코틀린 예제를 살펴보겠습니다.

위 예제처럼 코틀린은 함수를 변수에 할당할 수 있으므로 함수를 일급 객체로 취급합니다. 그럼 자바 예제를 살펴볼까요?

코틀린은 함수를 함수의 인자로 전달할 수 있고 함수의 반환값으로 반환할 수 있습니다. 반면 자바는 함수를 함수의 인자로 전달할 수 없고 함수의 반환값으로 반환할 수도 없습니다.

함수형 프로그래밍 언어 (Functional Programming)

함수형 프로그래밍 언어는 함수를 일급 객체로 간주하여 변수에도 할당할 수 있고, 함수의 인자로 전달할 수 있으며, 함수의 반환값으로 반환할 수 있는 언어를 의미합니다.

마치며

이번 포스트에서는 일급 객체와 함수형 프로그래밍에 대해 알아봤습니다. 다음 포스트에서는 람다식(익명 함수)과 함수형 인터페이스에 대해 알아보겠습니다.

어노테이션 프로세싱

어노테이션 프로세싱은 자바 컴파일러가 컴파일 단계에서 유저가 정의한 어노테이션을 분석하고 처리하는 기법입니다.

자바 컴파일러와 어노테이션 프로세서

자바 컴파일러(javac)는 자바 소스코드(.java)를 자바 바이트코드(.class)로 변환합니다. 이때 어노테이션 프로세서(Annotation Processor)를 플러그인 형태로 자바 컴파일러에 등록하여 사용합니다.

자바 컴파일러는 등록된 어노테이션 프로세서가 모두 처리될 때까지 반복하여 실행합니다. 이 때 어노테이션 프로세서에 구현된 로직에 따라 경고나 에러를 발생시키기도, 추가적인 자바코드를 추가하기도 합니다. 자바 컴파일러는 이렇게 최종적으로 완성된 자바 소스코드를 자바 바이트코드로 변환합니다.

어노테이션

자바로 개발을 하다 보면 클래스, 메서드, 변수 앞에 @Override 같은 @ 표시를 많이 봤을 것입니다. @를 어노테이션(Annotation)이라고 합니다.

어노테이션은 컴파일 타임이나 런타임에 특정 처리를 하도록 컴파일러에게 정보를 제공합니다. 이를 통해 소스코드 작성 단계에서 보일러코드를 제거하고 소스코드를 간결하게 작성할 수 있습니다. 컴파일 과정에서 어노테이션을 처리하여 데이터를 검증할 수 있으며, 소스코드를 추가하거나 변경할 수도 있기 때문입니다. 어노테이션은 크게 세 가지 용도로 사용됩니다.

  1. 컴파일 과정에서 에러나 경고를 감지하도록 컴파일러에게 정보를 제공
  2. 컴파일 과정에서 특정 코드나 파일을 생성/변경하도록 지시하도록 컴파일러에게 정보를 제공
  3. 런타임에서 특정 작업을 실행하도록 정보를 제공

타입 소거

타입 소거(Type Erasure)는 제네릭의 타입 파라미터를 구체화된 자료형으로 대체하는 작업을 의미합니다. 자바 컴파일러는 소스 코드를 컴파일할 때 타입 소거를 수행합니다.

타입 파라미터가 Unbounded Type인지 Bounded Type인지에 따라 소거 방식이 다릅니다.

(1) Unbounded Type의 타입소거

Unbounded Type은 형식 타입 파라미터로 전달받게될 실 타입 파라미터의 범위를 제한하지 않은 것을 의미합니다. <?>, <T> 형태의 타입 파라미터라고 생각하시면 됩니다.

와일드 카드(Wildcard)는 제네릭 클래스의 인스턴스를 함수의 인자로 전달할 때, 그 인스턴스의 타입 파라미터를 제한하는 것을 의미합니다. 개념이 어려우므로 예제를 통해 살펴보겠습니다.

와일드 카드<?>의 종류

  • <?> 타입 매개변수에 제한 없습니다.

2018/02/02 카테고리 Java

  3. 제너릭 함수(Generic Function)
  4. 제너릭의 와일드 카드(Wildcard)
  5. 타입 소거(Type Erasure)

제네릭 클래스

제네릭은 클래스에서 사용할 수 있습니다. 제네릭 클래스(Generic Class)는 다음과 같이 정의합니다.

<>안에는 아직 아직 타입이 결정되지 않음을 의미하는 알파벳 대문자를 넣어줍니다. 이를 타입 파라미터(Type Parameter)라고 합니다. 타입 파라미터는 관습적으로 대문자 E(Element), T(Type), K(Key), V(Value)등을 사용합니다.

이렇게 작성된 타입 파라미터에는 어떠한 타입의 데이터도 들어올 수 있습니다. 우선 타입 파라미터로 Integer를 전달해보겠습니다.

제네릭 클래스를 정의할 때 사용한 E를 형식 타입 파라미터(Formal type parameter)라고 합니다. 객체를 생성할 때 전달한 자료형은 Integer는 실 타입 파라미터(Actual type parameter)라고 합니다.

컴파일러는 타입 파라미터로 전달된 자료형을 클래스 내부의 T로 대체되는 것처럼 인식합니다. 마치 아래 코드와 같이 인식하는 것입니다.

실 타입 파라미터로 Integer뿐만 아니라 String, Person등 다양한 자료형을 전달할 수 있습니다.

제너릭 인터페이스와 제너릭 타입

제너릭은 클래스 뿐만 아니라 인터페이스와도 사용할 수 있습니다. 이를 제너릭 인터페이스(Generic Interface)라고 합니다.

제너릭 클래스와 제너릭 인터페이스를 합쳐서 제너릭 타입(Generic Type)이라고 합니다.

제너릭 타입(Generic Type)과 로 타입(Raw Type)

참고로 제너릭을 사용하면 제너릭 타입에 대응하여 로 타입(Raw Type)이라는 개념이 등장합니다. 로 타입(Raw Type)은 제너릭을 사용할 때 타입 파라미터를 사용하지 않는 것을 의미합니다.

위 코드에서 Box<String>는 제너릭 타입입니다. 반면 Box는 로 타입입니다.

제네릭의 큰 장점 중 하나는 컴파일 타임에 타입을 체크하여 타입 안정성을 확보한다는 것입니다. 따라서 제네릭을 사용할 때 로 타입의 사용을 권장하지 않습니다.

제한된 타입 파라미터 (Bounded Type Parameter)

키워드 extends를 사용하면 타입 파라미터로 들어오는 데이터의 타입을 제한할 수 있습니다.

<T extends Student>가 타입 파라미터를 제한하는 부분입니다. Student클래스와 이를 상속하는 클래스만 들어갈 수 있습니다.
