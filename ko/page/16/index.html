와일드 카드

와일드 카드(Wildcard)는 제네릭 클래스의 인스턴스를 함수의 인자로 전달할 때, 그 인스턴스의 타입 파라미터를 제한하는 것을 의미합니다. 개념이 어려우므로 예제를 통해 살펴보겠습니다.

와일드 카드<?>의 종류

  • <? extends T> 와일드 카드의 상한 제한(upper bound) - T와 그 자손들을 구현한 객체들만 타입 매개변수로 올 수 있습니다.
  • <? super T> 와일드 카드의 하한 제한(lower bound) - T와 그 조상들을 구현한 객체들만 타입 매개변수로 올 수 있습니다.
  • <?> 타입 매개변수에 제한 없습니다.

[자바] 제너릭 클래스(Generic Class)

2018/02/02 카테고리 Java

자바(Java)에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

다섯 개의 포스트에 걸쳐 제너릭(Generic)에 대해 살펴보고있습니다.

  4. 제너릭의 와일드 카드(Wildcard)

제네릭 클래스

제네릭은 클래스에서 사용할 수 있습니다. 제네릭 클래스(Generic Class)는 다음과 같이 정의합니다.

<>안에는 아직 아직 타입이 결정되지 않음을 의미하는 알파벳 대문자를 넣어줍니다. 이를 타입 파라미터(Type Parameter)라고 합니다. 타입 파라미터는 관습적으로 대문자 E(Element), T(Type), K(Key), V(Value)등을 사용합니다.

이렇게 작성된 타입 파라미터에는 어떠한 타입의 데이터도 들어올 수 있습니다. 우선 타입 파라미터로 Integer를 전달해보겠습니다.

제네릭 클래스를 정의할 때 사용한 E를 형식 타입 파라미터(Formal type parameter)라고 합니다. 객체를 생성할 때 전달한 자료형은 Integer는 실 타입 파라미터(Actual type parameter)라고 합니다.

컴파일러는 타입 파라미터로 전달된 자료형을 클래스 내부의 T로 대체되는 것처럼 인식합니다. 마치 아래 코드와 같이 인식하는 것입니다.

실 타입 파라미터로 Integer뿐만 아니라 String, Person등 다양한 자료형을 전달할 수 있습니다.

제너릭 인터페이스와 제너릭 타입

제너릭은 클래스 뿐만 아니라 인터페이스와도 사용할 수 있습니다. 이를 제너릭 인터페이스(Generic Interface)라고 합니다.

제너릭 클래스와 제너릭 인터페이스를 합쳐서 제너릭 타입(Generic Type)이라고 합니다.

제너릭 타입(Generic Type)과 로 타입(Raw Type)

참고로 제너릭을 사용하면 제너릭 타입에 대응하여 로 타입(Raw Type)이라는 개념이 등장합니다. 로 타입(Raw Type)은 제너릭을 사용할 때 타입 파라미터를 사용하지 않는 것을 의미합니다.

위 코드에서 Box<String>는 제너릭 타입입니다. 반면 Box는 로 타입입니다.

제네릭의 큰 장점 중 하나는 컴파일 타임에 타입을 체크하여 타입 안정성을 확보한다는 것입니다. 따라서 제네릭을 사용할 때 로 타입의 사용을 권장하지 않습니다.

제한된 타입 파라미터 (Bounded Type Parameter)

키워드 extends를 사용하면 타입 파라미터로 들어오는 데이터의 타입을 제한할 수 있습니다.

<T extends Student>가 타입 파라미터를 제한하는 부분입니다. Student클래스와 이를 상속하는 클래스만 들어갈 수 있습니다.

제네릭

제네릭(Generic)은 클래스나 메서드 내부에서 사용할 데이터의 자료형을 외부에서 설정하는 것입니다. 제네릭을 사용하면 다음과 같은 장점이 있습니다.
  1. 특정 타입에 종속되지 않은 유연한 로직
  2. 타입 안정성
  3. 자동 형변환

제네릭을 사용하면 특정 타입에 종속되지 않은 유연한 로직을 구현할 수 있습니다.

이제 Integer타입의 데이터를 추가하려고 합니다. 그러나 위 Stack클래스를 사용할 수는 없습니다. Stack클래스는 String타입의 데이터만 입력받고 저장하고 삭제할 수 있기 때문입니다. 따라서 별도의 StackForInteger클래스를 정의해야합니다.

그리고 타입 파라미터가 붙어있는 클래스를 제너릭 클래스라고 합니다. 제너릭 클래스는 인스턴스를 생성할 때 자료형 정보를 타입 파라미터에 전달합니다.

우선 타입 파라미터로 String을 전달하여 Stack을 생성해봅시다.

이제 String타입의 데이터를 추가할 수 있습니다.

(장점 2) 타입 안정성

타입 안정성은 런타임에 발생할 에러를 컴파일 타임에 찾아낼 수 있음을 의미합니다.

이제 CustomArrayList의 인스턴스를 생성하고 정수형 데이터를 넣어보겠습니다.

형 변환이 제대로 이루어지지 않았다는 오류 메시지입니다.

또한 잘못된 타입의 데이터를 입력하면 컴파일 시점에 오류를 탐지하여 타입 안정성을 높여줄 수 있습니다.

(장점 3) 자동 형변환

제네릭을 사용하면 자동으로 형변환을 해준다는 장점이 있습니다. 제네릭을 사용하지 않은 CustomArrayList클래스를 다시 살펴봅시다.

제네릭을 사용하면 별도의 형 변환이 필요없습니다.

2018/01/27 카테고리 Java

java.lang 패키지

java.lang 패키지는 자바에서 가장 기본적인 동작을 수행하는 클래스들의 집합입니다. java.lang 패키지에 있는 클래스는 import하지 않아도 바로 사용할 수 있습니다.

Object 클래스

Object클래스는 모든 자바 클래스의 최고 조상 클래스입니다. 자바의 모든 클래스는 암시적으로든 명시적으로든 Object클래스를 상속받고 있습니다. Object클래스는 java.lang패키지에 포함되며, Object클래스는 여러 메소드를 제공하고 있습니다.

[자바] 접근 제한자(public, private, protected, default)

2018/01/26 카테고리 Java

자바(Java)에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

이전 포스트에서는 형 변환(Type Casting)에 대해 알아봤습니다. 이번 포스트에서는 접근 제한자(Access Modifier)에 대해 알아보겠습니다.

접근 제한자

접근 제한자는 외부에서 클래스 내부의 메소드나 멤버변수에 접근하는 것을 제한하는 것입니다. 자바는 private, protected, default, public이라는 네 개의 접근 제한자를 지원합니다. 접근 제한자는 클래스, 생성자, 메소드, 멤버변수 앞에 붙일 수 있습니다.

(1) private

private은 같은 클래스 내에서만 접근이 가능합니다. 클래스에는 붙일 수 없으며 멤버변수, 메소드, 생성자 앞에만 붙일 수 있습니다.

(2) default

접근 제한자를 생략하면 기본적으로 default 접근 제한자가 적용됩니다. default는 동일한 패키지 내에서만 접근을 허용합니다. 멤버변수, 메소드, 생성자 뿐만 아니라 클래스 앞에도 붙일 수 있습니다.

Person클래스는 com.yologger.person패키지에 정의되어있고 접근 제한자는 default입니다.

따라서 다른 패키지인 com.yologger.app에서 접근할 수 없습니다.

(3) protected

protected는 같은 패키지에서 상속 관계에 있는 클래스에서만 접근을 허용합니다. 클래스에는 붙일 수 없으며 멤버변수, 메소드, 생성자 앞에만 붙일 수 있습니다.

(4) public

public은 다른 패키지라도 어디에서든 접근이 가능합니다.

다른 패키지인 com.yologger.app에서도 접근할 수 있습니다.

마치며

이번 포스트에서는 Java의 접근 제한자(Access Modifier)에 대해 알아봤습니다. 다음 포스트에서는 Object 클래스에 대해 알아보겠습니다.
