서명된 릴리즈 APK 파일

구글 플레이스토어에 앱을 출시하는 과정은 다음과 같습니다.

  1. 디버그 모드 대신 릴리스 모드로 변경
  2. 개인 키를 사용하여 앱에 서명하고 빌드하여 APK 파일 생성
  3. 구글 플레이스토어에 서명된 APK 파일 업로드

(1) 디버그 모드

애플리캐이션을 개발할 때는 디버그 모드로 앱을 빌드하고 실행합니다.

안드로이드 스튜디오는 디버그 모드일 때 디버그용 키를 사용하여 앱을 자동으로 서명합니다. 따라서 에뮬레이터나 연결된 실제 기기에 앱을 설치하고 실행할 때 별도의 서명 정보를 입력할 필요가 없습니다.

안드로이드 스튜디오는 프로젝트를 처음 실행하면 디버그용 키를 자동으로 생성합니다. 이 키가 저장된 키 스토어의 위치는 $HOME/.android/debug.keystore입니다.

디버그용 키는 설계상 보안이 유지되지 않습니다. 따라서 구글 플레이스토어를 포함한 대부분의 앱 스토어는 디버그용 키로 서명한 앱을 게시하는 것을 허용하지 않습니다.

(2) 릴리즈 모드

플레이스토어에 앱을 출시하려면 서명된 APK 파일을 업로드 해야합니다. 서명된 APK 파일을 생성하려면 빌드 타입(BuildType)를 릴리즈 모드로 변경해야합니다.

(3) 키스토어 및 키 생성

앱을 빌드하고 개인 키로 서명하면 서명된 APK 파일이 생성됩니다. 이 개인 키가 저장되는 곳을 키 스토어(Key Store)라고 합니다.

Build > Generate Signed APK 메뉴로 들어갑니다.

대화 상자에서 APK를 선택합니다.

먼저 키 스토어(Key Store)를 생성합시다. Create new버튼을 클릭합니다. 만약 이전에 키 스토어를 만들었다면 Choose existing버튼을 클릭하여 기존의 키 스토어를 사용합니다.

New Key Store 대화상자는 두 개의 섹션으로 나뉘어 있습니다. 첫 번째 섹션은 키 스토어 파일과 관련있습니다. 키 스토어 파일의 위치와 비밀번호를 입력합니다.

두 번째 섹션은 키 스토어에 저장될 개인 키와 관련있습니다. 별칭(Alias), 비밀번호, 유효기간은 필수로 입력해야합니다. 나머지 필드는 최소 하나의 필드에 정보를 입력합니다.

간혹 다음과 같은 경고 문구가 발생합니다. 특정 형식의 사용을 권장하는 경고문인데 서명하는데 아무 지장없으므로 무시합니다.

정보가 제대로 입력되었는지 확인하고 Next버튼을 누릅니다.

Destination Folder에서는 APK 파일이 생성될 경로를 지정합니다. 기본값은 프로젝트의 app폴더입니다. Build Variants는 release를 선택합니다. Signature Version은 V2를 선택하고 Finish버튼을 누릅니다.

이제 app폴더를 확인해봅시다. [your_project]/app/release 폴더 안에 서명된 APK 파일이 생성되었습니다.

2018/12/18 카테고리 Android

안드로이드(Android)에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

APK

APK(Android Application Package)는 안드로이드 어플리캐이션 설치 파일입니다. 안드로이드 시스템에서 APK 파일을 실행하여 어플리케이션을 설치할 수 있습니다. Windows 계열의 운영체제에서 인스톨러 역할을 한다고 보시면 됩니다.

빌드 프로세스

APK 파일은 다음과 같은 과정을 거쳐 생성됩니다.

  1. 안드로이드 프로젝트는 소스코드 및 리소스로 구성된 모듈과 의존성으로 구성됩니다.
  2. 그래들는 이 모든 것을 빌드하여 DEX 파일(Dalvik Executable File)과 컴파일된 리소스를 생성합니다.
  3. APK Manager는 개인키로 서명하여 APK 파일를 생성합니다.
  4. APK 파일은 구글 플레이스토어에 업로드하거나 안드로이드 시스템에 직접 설치할 수 있습니다.

APK 위치

디버그 모드에서 앱을 빌드하면 아래 경로에 APK 파일이 생성됩니다

왜 빌드 변형이 필요할까?

안드로이드에서는 하나의 프로젝트를 여러 버전의 앱으로 빌드할 수 있습니다. 그렇다면 왜 여러 버전의 앱으로 빌드를 해야할까요?

(1) 스마트폰 앱 vs. 태블릿 앱

스마트폰용 앱과 태블릿용 앱을 출시한다고 가정합시다. 두 앱은 기능은 동일하고 레이아웃만 약간 다릅니다. 이러한 경우 두 개의 프로젝트를 생성하여 각각 관리한다면 매우 비효율적일 것입니다.

(2) 무료 앱 vs. 유료 앱

이번엔 무료 앱과 유료 앱을 출시한다고 가정합시다. 두 앱은 레이아웃은 동일하며, 기능만 약간 다릅니다. 이러한 경우에도 두 개의 프로젝트를 생성하는 것은 비효율적입니다.

(3) 개발용 앱 vs 출시용 앱

마지막으로 개발용 앱과 출시용 앱이 있다고 가정합시다. 프로가드(ProGuard)는 앱 용량을 축소하고 코드를 난독화합니다. 다만 프로가드를 적용하면 빌드 시간이 길어지는데 굳이 디버그용 앱에 적용할 필요가 있을까요? 이러한 경우 출시용 앱에만 적용되도록 할 수 있습니다.

빌드 변형

하나의 프로젝트가 여러 버전으로 빌드되면 각각의 앱을 빌드 변형(Build Variant)이라고 합니다. 빌드 변형을 만드는 방법에는 두 가지가 있습니다.

  • 빌드 타입(Build Type)
  • 프로덕트 플래이버(Product Flavor)

안드로이드 스튜디오 프로젝트를 생성하면 debug, release라는 두 개의 빌드 타입(BuildType)이 생성합니다.

안드로이드 스튜디오 > 왼쪽 하단의 Build Variant 탭에서 빌드 타입을 확인할 수 있습니다.

각 빌드 타입에 대한 설정은 모듈 수준의 build.gradle의 buildTypes 섹션에서 할 수 있습니다. 프로젝트를 처음 생성하면 아마 아래와 같은 모습일 것입니다.

[안드로이드] 그래들 - (7) 모듈(Module)이란?

2018/12/07 카테고리 Android, Gradle

안드로이드(Android)에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

우리는 이전 포스트에서 의존성(Dependency)에 대해 알아봤습니다. 이번 포스트에서는 그래들의 모듈(Module)에 대해 알아보겠습니다.

모듈

Android Developers에서는 모듈(Module)을 다음과 같이 정의하고 있습니다.

모듈은 소스 파일 및 빌드 설정으로 구성된 모음이며, 이를 통해 프로젝트를 별개의 기능 단위로 분할할 수 있습니다. 프로젝트에는 하나 이상의 모듈이 포함될 수 있으며, 하나의 모듈이 다른 모듈을 종속 항목으로 사용할 수 있습니다. 각 모듈은 개별적으로 빌드, 테스트 및 디버그할 수 있습니다.

프로젝트와 모듈을 직접 만들어보면서 모듈이 무엇인지 알아보겠습니다.

모듈 생성하기

안드로이드 스튜디오 프로젝트를 생성하면 app이라는 모듈이 자동으로 생성됩니다.

이제 새로운 모듈을 추가해보겠습니다. Android Studio > File > New > New Module을 선택하면 다양한 모듈을 선택할 수 있습니다. 일단 Java or Kotlin Module을 선택합시다.

멀티 모듈의 장점

위 예제처럼 하나의 프로젝트에서 여러 개의 모듈을 사용하는 것을 멀티 모듈이라고 합니다.

각 모듈은 자신만의 build.gradle파일을 가지고 있습니다. 이는 각 모듈 단위로 빌드할 수 있다는 뜻입니다. 이를 통해 빌드 시간을 단축시킬 수 있습니다. 또한 기능별로 코드를 분리하여 중복을 제거하고 재사용성을 높일 수 있습니다. 뿐만 아니라 모듈 단위로 테스트를 할 수도 있어 테스트 용이성이 높아집니다.

모듈의 종류

자주 사용하는 모듈은 크게 세 가지입니다.

(1) Phone & Tablet Module

Phone & Tablet Module Module을 빌드하면 안드로이드 운영체제에서 실행 가능한 APK(Android Application Package)파일이 생성됩니다. 똑같은 앱을 스마트폰, 태플릿 용도로 나누어 출시한다던가 유료 앱, 무료 앱을 구분하여 출시할 때 사용할 수 있습니다. 프로젝트를 생성하면 기본으로 생성되는 app모듈이 바로 Phone & Tablet Module Module입니다.
