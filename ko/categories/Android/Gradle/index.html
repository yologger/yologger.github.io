2018/12/14 카테고리 Android, Gradle

Android의 특징과 개념에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

이전 포스트에서 .properties 파일에 대해 알아봤습니다. 이번 포스트에서는 빌드 타입(BuildType)에 대해 알아보겠습니다.

왜 빌드 변형(Build Variant)이 필요할까?

안드로이드에서는 하나의 프로젝트를 여러 버전의 앱으로 빌드할 수 있습니다. 그렇다면 왜 여러 버전의 앱으로 빌드를 해야할까요?

스마트폰용 앱과 태블릿용 앱을 출시한다고 가정합시다. 두 앱은 기능은 동일하고 레이아웃만 약간 다릅니다. 이러한 경우 2개의 프로젝트를 생성하여 각각 관리한다면 매우 비효율적일 것입니다.

이번엔 무료 앱과 유료 앱을 출시한다고 가정합시다. 두 앱은 레이아웃은 동일하며, 기능만 약간 다릅니다. 이러한 경우에도 2개의 프로젝트를 생성하는 것은 비효율적입니다.

마지막으로 디버그용 앱과 출시용 앱이 있다고 가정합시다. ProGuard는 앱 용량을 축소하고 코드를 난독화합니다. 다만 ProGuard를 적용하면 빌드 시간이 길어지는데 굳이 디버그용 앱에 ProGuard를 적용할 필요가 있을까요? 이러한 경우 출시용 앱에만 적용되도록 할 수 있습니다.

빌드 변형(Build Variant)

하나의 프로젝트가 여러 버전으로 빌드되면 각각의 앱을 빌드 변형(Build Variant)이라고 합니다. 빌드 변형을 만드는 방법에는 빌드 타입(Build Type)과 프로덕트 플래이버(Product Flavor)가 있습니다. 이번 포스트에서는 빌드 타입에 대해 알아보겠습니다.

[안드로이드] 그래들 - (7) 모듈(Module)이란?

2018/12/07 카테고리 Android, Gradle

Android의 특징과 개념에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

우리는 이전 포스트에서 의존성(Dependency)에 대해 알아봤습니다. 이번 포스트에서는 그래들의 모듈(Module)에 대해 알아보겠습니다.

개념

Android Developers에서는 모듈(Module)을 다음과 같이 정의하고 있습니다.

모듈은 소스 파일 및 빌드 설정으로 구성된 모음이며, 이를 통해 프로젝트를 별개의 기능 단위로 분할할 수 있습니다. 프로젝트에는 하나 이상의 모듈이 포함될 수 있으며, 하나의 모듈이 다른 모듈을 종속 항목으로 사용할 수 있습니다. 각 모듈은 개별적으로 빌드, 테스트 및 디버그할 수 있습니다.

프로젝트와 모듈을 직접 만들어보면서 모듈이 무엇인지 알아보겠습니다.

모듈 생성하기

안드로이드 스튜디오에서 프로젝트를 생성하면 app이라는 모듈이 자동으로 생성됩니다.

이제 새로운 모듈을 추가해보겠습니다. File > New > New Module을 클릭하면 여러 종류의 모듈을 선택할 수 있습니다. 일단 Java or Kotlin Module을 선택합시다.

모듈의 이름은 test_module입니다.

새로운 모듈 test_module이 추가된 것을 확인할 수 있습니다.

주목할 점은 모듈 test_module에 대한 build.gradle파일이 별도로 생성되었다는 것입니다.

모듈 test_module의 TestModule클래스는 다음과 같습니다.

모듈 사용하기

이제 모듈 app에서 모듈 test_module의 TestModule클래스를 사용하고자 합니다. 이때는 모듈 app의 build.gradle 파일에 의존성을 추가해야합니다.

build.gradle(Module: my_project.app)

이제 모듈 app에서 TestModule을 사용할 수 있습니다.

멀티 모듈의 장점

위 예제처럼 한 프로젝트에서 여러 개의 모듈을 사용하는 것을 멀티 모듈이라고 합니다.

위 예제를 보면 모듈 app에 대한 build.gradle파일, 모듈 test_module에 대한 build.gradle파일이 각각 존재하는 걸 알 수 있습니다. 이는 각 모듈을 별도로 빌드할 수 있다는 뜻입니다. 이를 통해 빌드 시간을 단축시킬 수 있습니다.

또한 기능별로 코드를 분리하여 중복을 제거하고 재사용성을 높일 수 있습니다. 뿐만 아니라 모듈을 각각 테스트를 할 수도 있어 테스트 용이성이 높아집니다.

모듈의 종류

자주 사용하는 모듈에는 크게 3가지 종류가 있습니다.

(1) Phone & Tablet Module

Phone & Tablet Module 모듈을 빌드하면 안드로이드 운영체제에서 실행 가능한 .apk 파일(Android Application Package)이 생성됩니다. 똑같은 앱을 스마트폰, 태플릿 용도로 나누어 출시한다던가 유료 앱, 무료 앱을 구분하여 출시할 때 사용할 수 있습니다. 프로젝트를 생성하면 기본으로 생성되는 app 모듈이 바로 Phone & Tablet Module입니다.

build.gradle 파일

그래들은 안드로이드 프로젝트를 빌드하는 빌드 도구입니다. build.gradle은 그래들 설정 파일입니다. 그래들은 build.gradle의 설정을 참고하여 프로젝트를 빌드합니다.

안드로이드 프로젝트를 생성하면 기본적으로 2개의 build.gradle 파일이 생성됩니다.

2. 모듈 수준의 build.gradle

모듈 수준의 build.gradle 파일은 특정 모듈에만 적용되는 build.gradle 파일입니다.

우선 안드로이드 플러그인과 코틀린 플러그인의 사용을 선언하고 있습니다.

그 다음 android 섹션에서 빌드할 때 사용할 SDK와 빌드 도구의 버전을 정의하고 있습니다.

defaultConfig 섹션에 선언된 항목들은 빌드하는 동안 해당 모듈의 AndroidManifest.xml 파일로 생성뙤는 요소를 정의합니다.

buildType 섹션에는 어플리케이션을 어떻게 빌드할 지를 명시합니다.

빌드 타입(BuildType)에는 기본적으로 debug 모드와 release 모드가 있습니다. release 모드로 빌드될 때는 ProGuard를 적용한다는 의미입니다. debug 모드는 따로 적혀있지 않으므로 디폴트 설정이 적용됩니다. 이렇게 아무 설정을 명시하지 않아도 기본 설정이 적용되는 것을 CoC라고 합니다. CoC에 대한 자세한 내용은 아래에서 살펴보겠습니다.

dependencies 섹션은 외부 모듈이나 라이브러리를 사용할 때 정의합니다. 같은 프로젝트 내 다른 모듈, jar파일, 다른 개발자가 만든 라이브러리 등을 사용하려면 dependencies 섹션에 명시해야합니다.

CoC

그래들은 설정보다는 관례(CoC, Convention over Configuration)라는 개념을 구현하고 있습니다. 즉, 그래들은 사전에 정의된 기본 설정값을 가지고 있습니다. 따라서 모든 설정을 설정하지 않아도 기본 설정값이 적용됩니다. 이 덕분에 개발자는 모든 설정을 하나하나 설정할 필요가 없어집니다. 자신의 요구 사항에 맞지 않는 설정만 빌드 설정 파일에서 변경하면 됩니다.
