개발자 계정 만들기

구글 플레이스토어에 앱을 출시하려면 구글 개발자 계정을 만들어야 합니다. http://play.google.com/apps/publish/signup을 방문하여 구글 개발자 계정을 생성합니다. (현재는 미화로 25달러를 결제해야하며, 한번 결제 시 평생 이용할 수 있습니다.)

앱 생성하기

구글 플레이 개발자 콘솔 계정을 만들었다면 구글 플레이 콘솔(Google Play Console)에 접속합니다. 그리고 Create app 버튼을 클릭합니다.

App details의 항목들을 입력합니다.

  1. App name: 플레이 스토어에 출시되는 앱 이름
  2. Default language: 기본 언어
  3. App or game: 게임 앱인 경우 game, 그 외에는 App
  4. Free or paid: 무료 앱인 경우 free, 유료 앱인 경우 paid (무료 선택 시 추후에 유료로 변경할 수 없습니다.)

Declarations 항목들을 입력합니다.

개발자 프로그램 정책과 미국 수출법규에 동의한다는 내용입니다.

이제 오른쪽 하단의 create app 버튼을 클릭하면 앱이 생성됩니다. 생성된 앱은 왼쪽 사이드 바 메뉴에의 All apps를 선택하면 확인할 수 있습니다.

앱 설정하기

All apps에서 생성한 앱을 선택합니다.

왼쪽 사이드바 메뉴에 Dashboard를 클릭합니다.

Dashboard 페이지에서는 앱 출시를 위해 입력해야할 정보가 단계별로 기재되어 있습니다. 하나씩 선택하고 입력하면 됩니다.

또한 Dashboard에서는 각 단계의 진행상황을 확인할 수 있습니다.

1. Start testing now

2. Set up your app

다음 7개 항목을 입력해야합니다.

(a) App access


All functionality is available without special access를 체크하고 저장합니다. 특별한 접근 권한 없이 모든 이용이 가능하다는 뜻입니다.

(b) Ads


광고 여부를 체크합니다. 앱 내에 광고가 있으면 “Yes, my app contains adds”를, 광고가 없으면 “No, my app does not contain ads”를 선택하고 저장합니다.

(c) Content ratings

콘텐츠의 등급을 설정합니다. Start new questionaire를 클릭하여 설문을 시작합니다.

이 단계에서는 이메일 주소와 앱 카테고리, 앱의 적합성을 조사합니다.

모든 단계가 끝나면 각 국가에서의 등급을 확인할 수 있습니다.

(d) Target audience and content

앱을 사용할 연령층과 어린이 관심 유도 여부 등을 확인합니다. 설문을 모두 완료합시다.

(e) News Apps

뉴스 앱인지를 체크합니다.

(f) App category & Contact

앱의 카테고리을 입력합니다.

개발자 연락처를 입력하고, 외부 마케팅 여부를 클릭합니다.

(g) Main store listing

앱 스토어에 표시될 앱 정보를 입력합니다.

  • App name: 플레이 스토어에 표시되는 앱 이름
  • Short description: 간단한 앱 설명
  • Full description: 자세한 앱 설명

  • App icon: 플레이스토어에 표시되는 앱 아이콘 이미지 (512x512px)
  • Feature graphic: 출시된 앱을 공유했을 때 나타나는 섬네일 이미지 (1024x500px)

[Android] 앱 출시하기 - (1) 서명된 릴리즈 APK 생성

2018/12/23 카테고리 Android

Android의 특징과 개념에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

앞으로 3개의 포스트에 걸쳐 앱을 출시하는 방법에 대해 알아보겠습니다.

  1. 서명된 릴리즈 APK 생성 ⬅️ ⬅️ ⬅️
  2. Google Play Store에 출시하기
  3. 새로운 버전 출시하기

앱 출시 프로세스

구글 플레이 스토어에 앱을 출시할 때는 다음 과정을 거칩니다.

  1. 디버그 모드 대신 릴리스 모드로 변경
  2. 앱 최적화와 역공학 방지를 위한 추가 작업
  3. 개인 키를 사용하여 앱에 서명하고 빌드
  4. 구글 플레이 스토어에 서명된 APK 파일 업로드

1. 디버그 모드

개발 단계에서는 디버그 모드로 앱을 빌드하고 실행합니다.

안드로이드 스튜디오는 디버그 모드일 때 디버그용 키를 사용하여 앱을 자동으로 서명합니다. 따라서 에뮬레이터나 연결된 실제 기기에 앱을 설치하고 실행할 때 별도의 서명 정보 입력이 필요없습니다.

안드로이드 스튜디오는 프로젝트를 처음 실행하면 디버그용 키를 자동으로 생성합니다. 이 키가 저장된 키 스토어의 위치는 $HOME/.android/debug.keystore입니다.

디버그용 키는 설계상 보안이 유지되지 않습니다. 따라서 구글 플레이 스토어를 포함한 대부분의 앱 스토어는 디버그용 키로 서명한 앱을 게시하는 것을 허용하지 않습니다.

2. 릴리즈 모드

플레이 스토어에 앱을 출시하려면 서명된 APK 파일을 업로드 해야합니다. 서명된 APK 파일을 생성하려면 빌드 타입(BuildType)를 릴리즈 모드로 변경해야합니다.

3. 키스토어 및 키 생성

소스코드와 리소스 파일을 빌드하고 개인 키로 서명하면 서명된 APK 파일이 생성됩니다. 이 개인 키가 저장되는 곳을 키 스토어(Key Store)라고 합니다.

우선 키 스토어를 생성합시다. Build > Generate Signed APK 메뉴로 들어갑니다.

대화 상자에서 APK를 선택합니다.

먼저 키 스토어를 생성합시다. Create new버튼을 클릭합니다. 만약 이전에 키 스토어를 만들었다면 Choose existing버튼을 클릭하여 기존의 키 스토어를 사용합니다.

New Key Store 대화상자는 두 개의 색션으로 나뉘어 있습니다. 첫 번째 색션은 키 스토어 파일과 관련있습니다. 키 스토어 파일의 위치와 비밀번호를 입력합니다.

두 번째 색션은 키 스토어에 저장될 개인 키와 관련있습니다. 별칭(Alias), 비밀번호, 유효기간은 필수로 입력해야합니다. 나머지 필드는 최소 하나의 필드에 정보를 입력합니다.

간혹 다음과 같은 경고 문구가 발생합니다. jks 키 저장소의 형식의 사용을 권장하는 경고문인데 APK 파일에 서명하는데 전혀 지장없으므로 무시하셔도 됩니다.

정보가 제대로 입력되었는지 확인하고 Next버튼을 누릅니다.

Destination Folder에서는 APK 파일이 생성될 경로를 지정합니다. 기본값은 프로젝트의 app 폴더입니다. Build Variants는 release를 선택합니다. Signature Version은 V2를 선택하고 Finish 버튼을 누릅니다.

이제 app 폴더를 확인해봅시다. [your_project]/app/release 폴더 안에 서명된 APK 파일이 생성되었습니다.

마치며

이번 포스트에서는 서명된 APK 파일을 생성하는 법에 대해 알아보았습니다. 다음 포스트에서는 구글 플레이 스토어에 앱을 출시하는 방법에 대해 알아보겠습니다.

[Android] APK 파일이란?

2018/12/18 카테고리 Android

Android의 특징과 개념에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

APK

APK(Android Application Package)는 안드로이드 설치 파일입니다. 안드로이드 시스템에서 APK 파일을 실행하여 어플리케이션을 설치할 수 있습니다. Windows 계열의 운영체제에서 installer 역할을 한다고 보시면 됩니다.

빌드 프로세스

APK 파일은 다음과 같은 과정을 거쳐 생성됩니다.

  1. 안드로이드 프로젝트는 소스코드 및 리소스로 구성된 모듈과 의존성으로 구성됩니다.
  2. 컴파일러는 이 모든 것을 빌드하여 DEX 파일(Dalvik Executable File)과 컴파일된 리소스를 생성합니다.
  3. APK Manager 개인키로 서명하여 APK 파일를 생성합니다.
  4. APK 파일은 플레이스토어에 업로드하거나 안드로이드 시스템에 직접 설치할 수 있습니다.

2018/12/15 카테고리 Android, Gradle

Android의 특징과 개념에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

이전 포스트에서는 빌드 타입(BuildType)에 대해 알아봤습니다. 이번 포스트에서는 빌드 변형의 프로덕트 플래이버(Product Flavor)에 대해 알아보겠습니다.

Product Flavor

스마트폰용 앱과 태블릿용 앱을 출시한다고 가정합시다. 두 앱은 레이아웃만 약간 다르고 기능적으로는 차이가 없습니다. 이러한 경우 프로덕트 플래이버(Product Flavor)를 사용할 수 있습니다.

build.gradle에 프로덕트 플래이버 추가하기

우선 모듈 수준의 build.gradle을 열어 다음과 같이 두 개의 프로덕트 플래이버를 추가합니다.

이제 Sync Now를 클릭하여 동기화 후 왼쪽 하단의 Build Variant 탭을 확인해봅시다.

총 4개의 빌드 변형이 추가되었습니다. 그래이들은 Build Type과 Product Flavor에 따라 자동으로 빌드 변형을 생성하고 <product-flavor><Build-Type>로 빌드 변형의 이름을 지정합니다. 따라서

  • mobileDebug
  • mobileRelease
  • tabletDebug
  • tabletRelease

가 생성되었습니다.

applicationIdSuffix은 applicationId에 추가되는 접미사입니다. applicationId가 “com.yologger.example”이고 applicationIdSuffix가 “.phone”이면 이 빌드 변형의 applicationId는 com.yologger.example.phone이 됩니다.

versionNameSuffix은 versionName에 추가되는 접미사입니다. versionName이 “2.1.3”이고 versionNameSuffix가 “-tablet”이면 이 빌드 변형의 versionName은 “2.1.3-tablet”이 됩니다.

[Android] 그레이들 - (10) 빌드 변형(Build Variant) - BuildType

2018/12/14 카테고리 Android, Gradle

Android의 특징과 개념에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

이전 포스트에서 .properties 파일에 대해 알아봤습니다. 이번 포스트에서는 빌드 타입(BuildType)에 대해 알아보겠습니다.

왜 빌드 변형(Build Variant)이 필요할까?

안드로이드에서는 하나의 프로젝트를 여러 버전의 앱으로 빌드할 수 있습니다. 그렇다면 왜 여러 버전의 앱으로 빌드를 해야할까요?

스마트폰용 앱과 태블릿용 앱을 출시한다고 가정합시다. 두 앱은 기능은 동일하고 레이아웃만 약간 다릅니다. 이러한 경우 2개의 프로젝트를 생성하여 각각 관리한다면 매우 비효율적일 것입니다.

이번엔 무료 앱과 유료 앱을 출시한다고 가정합시다. 두 앱은 레이아웃은 동일하며, 기능만 약간 다릅니다. 이러한 경우에도 2개의 프로젝트를 생성하는 것은 비효율적입니다.

마지막으로 디버그용 앱과 출시용 앱이 있다고 가정합시다. ProGuard는 앱 용량을 축소하고 코드를 난독화합니다. 다만 ProGuard를 적용하면 빌드 시간이 길어지는데 굳이 디버그용 앱에 ProGuard를 적용할 필요가 있을까요? 이러한 경우 출시용 앱에만 적용되도록 할 수 있습니다.

빌드 변형(Build Variant)

하나의 프로젝트가 여러 버전으로 빌드되면 각각의 앱을 빌드 변형(Build Variant)이라고 합니다. 빌드 변형을 만드는 방법에는 빌드 타입(Build Type)과 프로덕트 플래이버(Product Flavor)가 있습니다. 이번 포스트에서는 빌드 타입에 대해 알아보겠습니다.
