[안드로이드] JVM, 달빅 가상머신, ART의 차이점

2019/03/02 카테고리 Android, Android Framework

안드로이드(Android)에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

이전 포스트에서는 DEX 파일과 APK 파일에 대해 알아봤습니다. 이번 포스트에서는 JVM, 달빅 가상머신, ART에 대해 알아보겠습니다.

JVM

JVM(Java Virtual Machine)은 자바 바이트코드를 실행하는 가상머신입니다.


자바 컴파일러는 자바 소스코드를 컴파일하여 자바 바이트코드를 생성합니다. JVM은 이 자바 바이트코드를 실행하는 주체입니다.

플랫폼에 적절한 JVM만 설치되어있으면 어떤 플랫폼에서든 자바 바이트코드를 실행할 수 있습니다.

DVM

DVM(Dalvik Virtual Machine)은 안드로이드 애플리캐이션을 실행하는 주체입니다.

구글은 자바 가상머신인 JVM을 사용할 수도 있었지만 오라클과의 라이센스 문제 때문에 DVM을 개발하여 안드로이드에 넣게 됩니다.

뿐만 아니라 모바일 기기는 베터리 수명, 컴퓨팅 능력, 메모리 등의 자원이 데스크탑 환경에 비해 열악합니다. 이 때문에 모바일 기기에 맞게 나온 가상머신이 바로 DVM 입니다.



자바 컴파일러가 자바 바이트코드를 생성하는 것까진 동일합니다. 이후 DEX 컴파일러가 자바 바이트코드를 DVM에서 실행 가능한 DEX(Dalvik Executable)파일로 변환합니다. 최종적으로 DEX 파일과 리소스 파일을 압축하여 APK(Android Application Package)을 생성하게 됩니다.

DEX 컴파일러는 오랜 시간에 걸쳐 발전해왔습니다.

(1) DX 컴파일러

D8 컴파일러가 출시되기 전까지 사용되는 DEX 컴파일러입니다.

(2) D8 컴파일러

구글이 2018년 출시한 새로운 DEX 컴파일러입니다. Android Studio 3.0 버전부터 사용할 수 있으며, Android Studio 3.1 버전부터 기본 DEX 컴파일러가 되었습니다. DEX 컴파일러는 Android Gradle Plugin에 포함되어있습니다.

(3) R8 컴파일러

구글이 2019년 출시한 새로운 DEX 컴파일러입니다. Android Stuido 3.4 버전부터 사용할 수 있습니다. R8 컴파일러는 ProGuard가 기존에 수행하던 코드 난독화 및 최적화 작업을 대신 수행합니다. 따라서 Android Gradle Plugin 3.4.0 버전부터는 ProGuard 대신 R8 컴파일러를 사용하여 코드 난독화와 최적화 작업을 수행할 수 있습니다.

ART

ART(Android Runtime)은 안드로이드에서 사용되던 기존 DVM의 한계점을 해결하기 위해서 구글에서 새로 개발한 런타임입니다.

DVM은 내부적으로 JIT(Just In Time) 컴파일러를 사용합니다. 이 컴파일러는 애플리캐이션을 실행할 때마다 자바 바이트코드를 기계어로 컴파일합니다. 이 덕분에 설치 파일은 작아졌으나 CPU, 메모리, 베터리 사용이 높아지게 되었습니다.

ART는 내부적으로 AOT(Ahead On Time) 컴파일러를 사용합니다. 이 컴파일러는 애플리캐이션을 설치할 때 자바 바이트코드를 기계어로 컴파일합니다. 따라서 설치 파일이 커지고 설치 시간이 오래 걸린다는 단점이 있으나 런타임에서 CPU, 메모리, 베터리 사용량이 낮아지고 퍼포먼스가 좋아지게 되었습니다.

[안드로이드] DEX 파일, APK 파일이란?

2019/03/01 카테고리 Android, Android Framework

안드로이드(Android)에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

들어가며

이전 포스트에서는 프로가드(ProGuard)에 대해 알아봤습니다. 이번 포스트에서는 DEX파일과 APK파일에 대해 알아보겠습니다.

APK

APK(Android Application Package)는 안드로이드 어플리캐이션 설치 파일입니다. 안드로이드 시스템에서 APK파일을 실행하여 어플리케이션을 설치할 수 있습니다. Windows 계열의 운영체제에서 인스톨러 역할을 한다고 보시면 됩니다.

빌드 프로세스

APK파일은 다음과 같은 과정을 거쳐 생성됩니다.

  1. 안드로이드 프로젝트는 소스코드 및 리소스로 구성된 모듈과 의존성으로 구성됩니다.
  2. 그래들는 이 모든 것을 빌드하여 DEX파일과 컴파일된 리소스를 생성합니다.
  3. APK Manager는 개인키로 서명하여 APK 파일를 생성합니다.
  4. APK 파일은 구글 플레이스토어에 업로드하거나 안드로이드 시스템에 직접 설치할 수 있습니다.

APK 위치

디버그 모드에서 앱을 빌드하면 아래 경로에 APK 파일이 생성됩니다

마치며

이번 포스트에서는 APK(Android Application Package)에 대해 알아봤습니다. 다음 포스트에서는 달빅 가상머신과 ART에 대해 알아보겠습니다.

앱 게시 취소

앱 게시가 취소되면 새로운 사용자는 플레이스토어에서 앱을 검색하거나 다운로드할 수 없습니다. 그러나 기존 사용자는 앱을 계속 사용할 수 있습니다.

우선 구글 플레이 콘솔(Google Play Console)에 접속합니다.

왼쪽 사이드바 메뉴 > All apps로 이동합니다. 이 곳에서 출시한 앱 목록을 확인할 수 있습니다. 삭제할 앱을 선택합니다.

Setup > Advanced settings로 이동합니다.

App availability를 Unpublished로 변경하고 Save changes버튼을 누르면 앱 게시가 취소됩니다.

변경 사항이 반영되는데 몇 시간이 걸릴 수 있습니다.

게시가 취소된 앱은 삭제되기 전까지 다시 출시할 수 있습니다.

앱 삭제

구글 플레이 콘솔 고객센터(Google Play Console Help)에 접속합니다.

지원이 필요한 사항을 입력합니다.

앱 삭제를 선택합니다.

이메일을 선택합니다.

충족 사항을 확인합니다.

이름, 성, 이메일 주소, 내 위치, 개발자 이름을 입력합니다.

개발자 계정 ID를 입력합니다. 계정 세부정보 페이지를 선택하면 개발자 계정 ID를 확인할 수 있습니다.


패키지 이름을 입력합니다.

패키지 이름은 구글 플레이 콘솔 > All apps에서 확인할 수 있습니다.

문제 해결에 도움이 되는 스크린샷을 추가합니다. 삭제할 앱의 스크린샷을 첨부합니다.

제출하면 처리되는데 1~2일 정도가 소요됩니다.

[안드로이드] 앱 출시 - (5) 플레이스토어에 앱 출시하기

2019/02/08 카테고리 Android, App Release

안드로이드(Android)에 대해 연재하고 있습니다. 목차는 이 곳에서 확인할 수 있습니다.

소프트웨어 출시 과정

소프트웨어는 출시되기 전 많은 테스트를 거칩니다.

(1) 알파 테스트

알파 테스트는 회사 내부에서 진행하는 테스트입니다. 내부 테스트라고도 하며 보통 개발팀이나 품질관리팀에서 실시합니다.

(2) 베타 테스트

베타 테스트는 회사 외부에서 진행하는 테스트입니다. 베타 테스트에는 두 종류가 있습니다.

  • 클로스드 베타 테스트: 회사가 특정 사용자들만 선발해서 진행하는 테스트입니다. 비공개 테스트라고도 합니다.

  • 오픈 베타 테스트: 정식 출시 이전에 모든 사용자에게 진행하는 테스트입니다. 공개 테스트라고도 합니다.

(3) 출시

소프트웨어가 모든 테스트를 통과하면 시장에 출시합니다.

안드로이드 내부 테스트

안드로이드에서는 알파 테스트보다 내부 테스트라는 용어를 더 많이 사용합니다. 이제 내부 테스트 절차에 대해 알아보겠습니다.

(1) 개발자 계정 만들기

구글 플레이스토어에 앱을 출시하려면 구글 개발자 계정을 만들어야 합니다. http://play.google.com/apps/publish/signup을 방여문하 구글 개발자 계정을 생성합니다. (현재는 미화로 25달러를 결제해야하며, 한번 결제 시 평생 이용할 수 있습니다.)

(2) 구글 플레이 콘솔에 앱 생성하기

구글 플레이 콘솔 > All apps > Create app을 선택합니다.

App details의 하위 항목을 입력합니다.


  • App name: 플레이 스토어에 출시되는 앱 이름
  • Default launguage: 기본 언어
  • App or game: 게임 앱인 경우 game, 그 외에는 App
  • Free or paid: 무료 앱인 경우 free, 유료 앱인 경우 paid (무료 선택 시 추후에 유료로 변경할 수 없습니다.)

Declarations의 하위 항목을 모두 체크합니다.


  • Developer Program Policies: 개발자 프로그램 정책에 동의
  • US export laws: 미국 수출법규에 동의

구글 플레이 콘솔 > All apps을 선택하면 생성한 앱을 확인할 수 있습니다.

(3) 앱 정보 입력

앱을 테스트하거나 출시하려면 앱에 대한 정보를 입력해야합니다. 구글 플레이 콘솔 > All apps에서 생성한 앱을 선택합니다.

왼쪽 사이드 바 메뉴에서 Dashboard를 선택합니다. Dashboard에서는 앱을 테스트하거나 출시하기위해 입력해야할 항목들을 확인할 수 있습니다. 항목은 크게 세 가지로 구성되어있습니다.

  • Start testing now
  • Set up your app
  • Release your app


View tasks를 누르면 아래 사진처럼 더 자세한 업무들을 확인할 수 있습니다.

완료된 업무는 옆에서 초록색 완료 아이콘을 확인할 수 있습니다.

(4) 테스터 선택

Start testing now항목에서 Select testers를 선택합니다.

내부 테스트는 최대 100명의 테스터를 선택할 수 있습니다. Feedback URL or email address에는 테스터로부터 피드백 받을 이메일을 입력합니다. 이후 Create email list을 클릭하여 테스터의 이메일을 입력합니다.

테스터의 이메일을 추가하고 Save changes를 선택합니다.

테스터의 이메일은 구글 계정만 가능합니다.

(5) 새로운 릴리즈 생성

Dashboard로 돌아와서 Start testing now > Create a new release를 선택합니다.

다시 한번 Create a new release를 선택합니다.

이전 포스트에서 생성한 서명된 APK파일을 업로드합니다.

Release name과 Release notes를 입력하고 Save버튼과 Review release버튼을 순서대로 클릭합니다.

항목이 제대로 입력되었는지 확인하고 Start rollout to internal testing버튼을 클릭합니다.

(6) 테스트 참여 URL

구글의 심사는 몇 시간에서 며칠이 소요될 수 있습니다. 심사가 끝나면 테스터 참여 URL이 발급되고 이를 전달할 수 있습니다. 구글 플레이 콘솔 > 왼쪽 메뉴 바 > All app에서 본인이 알파 출시한 앱을 선택합니다.

왼쪽 메뉴 바 > Testing > Internal testing으로 이동합니다.

Testers탭에서 하단 Copy link를 클릭하여 앱 설치 주소를 복사할 수 있습니다.

복사한 링크를 테스터에게 전달하면 테스터는 앱을 설치하고 테스트할 수 있습니다.

테스터 선택에서 추가한 구글 계정만이 앱을 설치하고 테스트할 수 있습니다.

마치며

이번 포스트에서는 알파 테스트(내부 테스트)에 대해 알아봤습니다. 다음 포스트에서는 클로스드 베타 테스트(비공개 테스트)에 대해 알아보겠습니다.

(b) Ads


광고 여부를 체크합니다. 앱 내에 광고가 있으면 “Yes, my app contains adds”를, 광고가 없으면 “No, my app does not contain ads”를 선택하고 저장합니다.

소프트웨어 출시 과정

소프트웨어는 출시되기 전 많은 테스트를 거칩니다.

(1) 알파 테스트

알파 테스트는 회사 내부에서 진행하는 테스트입니다. 내부 테스트라고도 하며 보통 개발팀이나 품질관리팀에서 실시합니다.

(2) 베타 테스트

베타 테스트는 회사 외부에서 진행하는 테스트입니다. 베타 테스트에는 두 종류가 있습니다.

  • 클로스드 베타 테스트: 회사가 특정 사용자들만 선발해서 진행하는 테스트입니다. 비공개 테스트라고도 합니다.

  • 오픈 베타 테스트: 정식 출시 이전에 모든 사용자에게 진행하는 테스트입니다. 공개 테스트라고도 합니다.

(3) 출시

소프트웨어가 모든 테스트를 통과하면 시장에 출시합니다.

안드로이드 비공개 테스트

안드로이드에서는 오픈 베타 테스트보다 공개 테스트라는 용어를 더 많이 사용합니다. 비공개 테스트는 테스트용 앱을 회사 외부의 불특정 다수에게 배포하고 피드백을 받습니다.

이제 공개 테스트에 대해 알아보겠습니다.
